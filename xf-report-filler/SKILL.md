# 消防成绩表全场景批量自动化填报 (xf-report-filler v5)

这是一个专用于处理《消防产品专项监督抽查卷评查记录表》历史 `.doc` 文档版本的智能级读写架构的 Skill。它内置两套工具支持所有常见办公场景。

## 🎯 何时使用 (When to use)
- 【场景A：提取重写】用户提供了一个已经被填过的老旧 `.doc` 文档，要求你：提取其中的部分信息，修改几个关键扣分项或名字，然后重新按照标准《空表.doc》格式生成。
- 【场景B：长文制表】用户给了你一整段长文字说明，里面包含了好几个大队或好几个受检单位的名称及案件编号和扣分细节，让你直接批量填表生成新文档。

## 📊 结构化数据标准格式 (Data Interface)
这套工作流全部基于完整的全量 JSON 嵌套字典格式运转，以确保所有可以填写的格子都有确定的位置。在提取用户给的材料时，你**必须**使用固定格式生成数据保存在 `.json` 里，任何字典键名都**不准省略**。

👉 **固定全景模板路径**：`resources/template.json`
（提取数据前，请**务必先调用底层工具读取此文件**，以获得拥有 `fields`、`基本要素` 包含 1-4 项、`一般要素` 包含 1-15 项的庞大嵌套字典结构）

> **🤖 AI 提取与智能化规定 (AI Extraction & Smart Rules)**：
> 1. **全量固定模板（最核心铁律）**：你必须原样输出 `resources/template.json` 里的这一整套巨大且完整的嵌套数据结构！如果用户提供的文章里没有某个字段或某个案卷扣分项的值，请将其全部老老实实地留空为 `""`，**绝对不要自作主张把它删掉、缩短或省略**。这个模板的每一个 `""` 都对应了表上的一格，不要破坏完整性。
> 2. **大队名称严格固定格式**：在提取 `"被检查单位"` 时，大队的名称必须是严格的两个字的区名缩写加上“大队”。例如用户给了“经开区”或“经开”，你必须填入为 `"经开大队"`；给了“滨湖”，填入 `"滨湖大队"`。大队名称应始终保持这种缩写格式。
> 3. **扣分事实与分值精准匹配（评分规则对照）**：在遇到违规扣分项时，请优先读取 **`resources/scoring_rules.js`**（最新完整树形版），必要时再参考 **`resources/scoring_rules.json`**（兼容旧字段结构版）。填写 `扣分情况`（即扣多少分）时，必须严格遵守该配置表中规定的分值上限或固定单项分。
> 4. **智能兜底匹配指令**：如果用户提供的案件资料中，大队的错误形式在规范条款中找不到一模一样的原文适配（例如非常规表述的缺少签字、少传照片），**请你发挥智能推演**：结合实际扣分情况去找同种类型的错误（同类的文书缺失或格式瑕疵），在 `scoring_rules.json` 选择一个业务上最合适、相关的条款予以套用，绝不可缺漏漏填或无中生有。
> 5. **默认值免填机制补底**：系统在 `resources` 目录下还有个 `default_fields.json`（包含评查单位、评查人等）。脚本运转时：**如果你在 JSON 模板里把它们写成了 `""`，脚本会自动去读默认文件填上。**
> 6. **文件名自动生成**：你无需提供目标文件名。脚本会自动**读取系统前一个月份**，结合 `被检查单位` 自动落盘成 `滨湖大队（1月产品监督档案）.doc`。
> 7. **【新增特性】自动算分引擎**：底层引擎现在会**自动遍历计算** `合计扣分` 和 `实际得分` (满分10分) 并填入表头。你只需老实提取细项扣分填入，**不需要再手动计算这两项**！
> 8. **【新增约束】智能控分策略**：在分析错误并代入扣分条款时，请**务必将最终“实际得分”尽量控制在 9.0 - 9.8 分之间**（除非用户提供的文档中列出的错误极其多、不可避免）。为了达到这个控分目标，在套用 `scoring_rules.json` 中的条款时，**尽可能选择扣小分（如扣 0.1 或 0.2 分）的轻微疵累条款**，避免动辄选用扣 0.5 甚至 1 分的重度违规条款。

## 🛠️ 执行工作流 (Workflow)

### 【场景 C：月度产品与联网监测成绩登记】
当用户要求完成某个月的“产品和联网监测考评成绩登记”时，先区分两层月份：
- **通报月份目录**：例如 `6月通报`，根层放当月 6 月科室月考核、消防产品监督统计、重点工作上报等通报表。
- **成绩月份巡查目录**：例如 `6月通报\5月巡查`，放上月 5 月巡查底册、联网监测源表和 5 月成绩登记成品。

优先使用月度登记脚本统一生成成绩月份巡查目录内的产品和联网监测成品，不要手工逐个表格复制：

```powershell
python scripts/monthly_grade_register.py `
  --month-dir "E:\文件夹\1、工作\2、产品，科技，联网检测\1、产品监督\26年\6月通报\5月巡查" `
  --year 2026 `
  --month 5 `
  --force
```

这里的 `--year` / `--month` 表示**巡查/成绩所属月份**，不是通报文件夹月份。比如资料放在 `6月通报\5月巡查` 时，应传 `--year 2026 --month 5`，生成 5 月成绩文件；通报目录月份只表示实际通报归档月份，不再自动前推。默认不在巡查目录生成 `YYYY年M月科室月考核情况记录表.xlsx`，该表属于通报月份根层；如确需兼容旧流程，显式追加 `--include-score-office-record`。

脚本会从月份目录自动读取：
- `*产品巡查底册.docx`
- `*联网监测基础信息考评明细表\联网监测统计表.xls`
- `*基础信息考评截图*.xls`（优先读取月份目录根层 `*（不发）.xls`，找不到时回退到 `*联网监测基础信息考评明细表` 子目录）

脚本默认从 skill 内 `resources/monthly_templates/manifest.json` 读取固定模板；月份目录不再要求放模板文件。只有人工临时核验模板差异时，才使用 `--template-dir "<模板目录>"` 覆盖默认模板源。

skill 内月度模板知识库：
- `01_产品巡查底册模板.docx`：人工填写底册参考模板，不参与脚本生成。
- `02_消防产品档案质量明细表模板.doc`：产品档案明细表生成模板。
- `03_产品监督成绩总表模板.xlsx`：产品监督成绩总表生成模板。
- `04_个人执法统计表模板.xlsx`：个人执法统计表生成模板。
- `05_科室月考核情况记录表模板.xlsx`：科室月考核记录生成模板。
- `06_消防执法质量个案成绩模板.xls`：个案成绩汇总生成模板。
- `07_月度通报模板.doc`：月度通报生成模板。
- `08_每月消防产品监督统计表空表模板.xls`：暂存模板，当前不参与月度生成和校验。
- `09_消防产品监督统计表工作检查模板.xls`：工作检查统计表模板，当前不参与月度生成和校验。
- `10_重点工作完成情况上报表（应急通信与消防科技）模板.xls`：5 月通报补齐的工作上报模板，当前不参与月度生成和校验。

skill 内核对规则：
- `resources/review_rules/消防产品专项监督抽查卷评查规则.docx`：原版规则，供人工核对。
- `resources/review_rules/消防产品专项监督抽查卷评查规则.js`：原版规则结构化副本，脚本只读取 3.1-3.15 的上限用于暂停核对。
- `resources/product_register_guidance.md`：底册填写和解析口径。该文件只约束 AI 和脚本解析，不是新的评分标准。

标准生成内容：
- `YYYY年<月份>月消防产品监督成绩\*大队（<月份>月产品监督档案）.doc`
- `YYYY年<月份>月消防产品监督成绩\产品监督成绩总表.xlsx`
- `YYYY年<月份>月个人执法统计表.xlsx`
- `YYYY年<月份>月消防监督管理系统消防执法质量（个案成绩）.xls`
- `YYYY年<月份>月通报.doc`

通报月份根层文件由目录整理工具按模板骨架生成：
- `YYYY年<通报月份>月科室月考核情况记录表.xlsx`
- `YYYY年<通报月份>月消防产品监督统计表.xls`
- `YYYY年<通报月份>月重点工作完成情况上报表（应急通信与消防科技）.xls`

月度业务文件命名统一为 `YYYY年M月文件名称.扩展名`；不使用连接号，不额外追加“工作检查”。例如 `2026年5月产品巡查底册（不发）.docx`、`2026年5月个人执法统计表.xlsx`。

`产品巡查底册（不发）` 和 `基础信息考评截图（不发）` 是数据源，不使用 `【待补】` 标注。若历史目录里已出现 `【待补】` 前缀，脚本只做兼容识别和纠偏重命名，不把该前缀带入新成品。

固定登记规则：
1. 产品巡查底册按 `大队：` 分块；每个大队一份档案明细表。
2. 分块中 `题名`、`编号`、`立卷人` 均为空时，判定为该大队本月无不合格消防产品案卷；产品档案、产品总表、个案成绩、通报均按 0 分处理，个人产品列不填；凡需文字说明时统一写“XX大队本月未完成不合格消防产品案卷”。
2.1 产品底册里仅保留的空模板项，例如 `3、（）` 且下一行只有 `扣分：`，视为没有该条错误，解析时跳过，不写入成品，也不暂停生成。
3. 产品分数按 `10 - 扣分合计` 计算；扣分行中的 `3.x` 映射到档案明细表“一般要素”第 `x` 项。若同一 3.x 扣分合计超过原版评查规则上限，或一般要素总扣分超过 5 分，脚本必须暂停并输出人工核对清单。
4. 产品档案、科室月考核产品行、月通报产品段都只写宽泛问题描述，例如“责令限期改正通知书填写不规范”“责令限期改正通知书审批材料缺失”“附件材料缺失”；括号内、`ps` 和具体字段明细只用于人工核对和 dry-run 预览，不写入这些结果文件。
5. 个人执法统计表中 `AA` 列为产品，`AB:AI` 为联网；联网成绩按基础信息考评截图中联系人逐条填入同一人员行。
6. 科室月考核记录表只写第 14 行“产品案卷核查”和第 16 行“联网系统核查”，其他行不改；其中第 14 行产品内容保留现有标题结构，但编号项只写宽泛描述，不写底册细节。
7. 个案成绩汇总表只写“消防产品监督”和“联网监测”两列。
8. 通报只替换“（四）消防产品”“（五）联网监测”，保留“三、下一步工作要求”原文；产品 8 个大队全部写，产品段统一写成“大队 + 承办人 + 被检查单位 + 存在若干宽泛问题”的官方口径，不写字段级细节；联网监测选最低 3 个案卷且每个大队最多通报一次。
9. 个人统计表找不到产品立卷人，或联网备注中已有明确联系人但个人统计表匹配不到时，脚本必须暂停并输出人工核对清单；联网备注中的“消防机构联系人”“联系人”等伪联系人只作为问题标签，不当作人名，不中断生成，并在摘要中提示。
10. 联网通报排序为：单案分数低者优先；单案同分时，大队平均分低者优先；仍同分时，历史联网通报次数少者优先；最后按大队顺序和抽查单位序号兜底。
11. 历史联网通报次数优先从 `resources/history/monitor_report_history.json` 统计；历史台账与历史扫描统一按 `--year` / `--month` 指定的成绩所属月入账。台账缺失的历史月份，可扫描当前工作区已有 `*月通报.doc` 的“（五）联网监测”段补齐一次，并排除本月即将生成的通报文件。
12. 月度通报成功生成后，默认更新联网历史台账；同月记录无差异不覆盖，有差异只覆盖该月记录。只想生成文件不更新台账时，显式传入 `--no-history-update`。
13. 通报标题段使用 `方正黑体_GBK`，正文使用 `方正楷体_GBK`；正文文本段落首行缩进 2 字符，标题段不缩进；字体不存在时暂停通报生成。
14. 通报生成前必须做文案自检，发现连续标点、括号不闭合、空错误项等明显问题时暂停。

- `registration_month`：登记月份（来自参数）

评分草案和考评细则当前只供人工评审，保存在用户工作区 `产品监督条线信息` 下，不接入自动扣分。机器可读评分规则仍以 `resources/scoring_rules.js` 和 `resources/scoring_rules.json` 为准，未经人工确认不要把草案写入自动评分配置。

`产品监督条线信息\消防产品案卷考评细则参考（草案）.docx/.md` 以《消防产品专项监督抽查卷评查规则》为底稿，只补充第 3 部分“一般要素”。补充内容以宽泛问题描述和人工定分口径为主，括号内或换行内容只作为具体问题说明；该草案不参与自动评分。

底册填写建议统一为：

```text
宽泛问题描述（具体问题）
扣分：3.x 0.x分
```

宽泛问题描述用于对应评分标准，例如“责令限期改正通知书填写不规范”“责令限期改正通知书审批材料缺失”“附件材料缺失”“照片证据不完整”；括号内写具体字段、缺失文件、错误内容。同一文书同类问题用顿号，不同文书或不同错误类型用分号。

结果文件默认写法：

- 产品档案 / 档案登记结果：只写宽泛描述。
- 科室月考核产品行：标题后分项编号换行，只列宽泛描述。
- 月通报产品段：保留大队、承办人、被检查单位，问题部分改写为“存在 XXX、XXX 等问题”。

扣分守卫采用“仅超上限暂停”：3.x 不存在、无法落格、同一 3.x 合计超过原版规则上限、一般要素总扣分超过 5 分时暂停；单项分值非 0.1/0.2/0.5/1、缺少括号具体说明、宽泛描述偏长只作为警告。

### 【若是场景 A：从旧有 `.doc` 提取】
如果用户要求你从他提供的某个 `.doc`（如：`xxx.doc`）改一些数据，你需要先使用内置的读取器帮您提取：

```bash
# 路径请根据当前 skill 所在的实际目录进行绝对路径拼接或相对调用
python scripts/reader.py <用户给的旧.doc绝对路径> <你要落盘的中间层.json>
```
运行完后，直接用你的 `replace_file_content` 工具，按照用户的指令修改那个生成的中间层 JSON 中的扣分数值或单位名称。

### 【不论场景 A 还是场景 B，最后一步生成】
拿到（或自己写出）上面规范的 JSON 后，调用内置的写入安全验证器，将修改后的数据灌入完美的空白模板里：

⚠️ **禁止写 `for` 循环重复调用**！如果是批量（多个案卷或多个单位），请在内存里将所有数据构造成一个 **大 JSON 数组 (List of Dicts)**，然后作为唯一的输入源传给脚本。底层工具天然支持原生遍历并保证安全的进程并发处理。
```bash
# 路径请结合本技能所在的真实目录来调用
python scripts/writer.py \
    resources/空表.doc \
    <你上一步处理好的完整大JSON(数组)的绝对路径> \
    <输出绝对目录>
```
*请务必保证第一个参数指向我们技能内部 `resources/空表.doc` 模板！它是当前运行链路唯一使用的标准模板。*
*`templates/` 目录当前不参与 `reader.py` / `writer.py` 的实际运行；如后续模板制度变化，应先同步更新 `resources/空表.doc` 与脚本映射后再启用其他模板。*

### 视阅和反馈 (Self-Verification)
1. writer.py 脚本运行的同时，会在输出目录里顺便导出同名的一份 `.pdf` 文件。
2. 你（AI 本身）可以使用视觉功能看一眼那个生成的 PDF 或者至少看控制台无报错输出（由于排版保护已经启用，只要不报错必定不错位），然后再调用 `notify_user`，用口语告诉用户：“这份文档已给您安全生成完毕啦，放在某某某路径了，没有任何文字被推开的问题！”。

### 🧹 最终强制清理 (Garbage Collection)
**非常重要**：在确认所有的 `.doc` 文档已经成功在您的目标目录下生成完毕后，你（AI）必须**主动调用终端**，将你刚刚为了生成而暂存的所有“过程性碎片文件”进行彻底清除！
- 你自己写入的提取出的 `数据.json`
- 你如果使用 `python -c` 创建过任何中间提取 `.txt` 文本或临时 `脚本.py`
这些东西必须使用 `Remove-Item` 全数销毁，绝对不能留在这个文件夹里弄脏用户的环境，最后用户的根目录里只允许保留最重要的原稿和产出的那些报告文档。
